# 🍗 QSR Sales Forecasting — *Technical Validation*

A **daily sales forecasting** system for a quick-service restaurant (QSR) chain,
focused on what actually earns a business team's trust: **validating the model
before relying on it.**

> ⚠️ **100% synthetic data.** Every dataset in this repository is generated by
> simulation (`data/generate_synthetic_data.py`) and does not come from any real
> company. The goal is to showcase the **methodology and technical approach**, not
> a third party's business data.

---

## 🎯 The problem

In a restaurant operation, forecasting sales isn't just "guessing the yearly
total." The real challenge is:

1. Forecast sales accurately enough to plan around them.
2. Break that annual number down into a **daily target per store** the operation
   can actually use (staffing, inventory, purchasing).
3. **Trust** the model: how do I know it isn't badly off?

This repository focuses on point 3, which is the foundation for everything else.

---

## 🧪 What this repo demonstrates: the *backtest*

Instead of asking the model to predict the future (which can't be verified), we
ask it to predict a year we **already know**:

- Train on **2021–2024**.
- Forecast **2025**, which the model has never seen.
- Compare the forecast against what actually happened.

### Results on the synthetic data

| Metric            | Result  |
|-------------------|---------|
| **Monthly MAPE**  | **0.88%** |
| Daily MAPE        | 3.34%   |
| Annual bias       | +0.14%  |

> A monthly MAPE below 1% means the model reproduces the chain's seasonal
> structure with high fidelity. (The exact figure depends on the generator's
> random seed.)

The notebook also produces a visual **actual vs. forecast** comparison:

📓 **[`notebooks/01_validacion_backtest.ipynb`](notebooks/01_validacion_backtest.ipynb)**

## 🔧 Methodology

- **Model:** [Prophet](https://facebook.github.io/prophet/) with **yearly** and
  **weekly** seasonality (the two dominant patterns in a QSR: time of year +
  strong weekend effect).
- **Validation:** temporal holdout (train 2021–2024 / test 2025), with MAPE and
  bias aggregated monthly — the granularity a budget is typically built at.
- **Synthetic data** with realistic structure: growth trend, yearly and weekly
  seasonality, holiday uplifts, and store-level idiosyncratic noise.

## 🚀 How to run it

```bash

git clone https://github.com/Gust-CM/qsr-sales-forecasting.git
cd qsr-sales-forecasting

# 2. Install dependencies
pip install -r requirements.txt

# 3. Generate the synthetic data
python data/generate_synthetic_data.py

# 4. Open the validation notebook
jupyter notebook notebooks/01_validacion_backtest.ipynb
```

## 🗺️ Roadmap (full project)

This repo covers the **validation** step. The complete methodology, applied in a
professional setting, also includes:

- Next-year forecasting with holidays, the commercial calendar, and recurring
  promotions as regressors.
- Distributing the annual budget into **daily targets per store**, factoring in
  payday cycles, special events, and zone sensitivity.
- Projecting **transactions** from the average ticket adjusted for pricing.
